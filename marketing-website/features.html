Features That Make Steps Unique

Designed from the ground up for teaching programming fundamentals

üèóÔ∏è Architectural Metaphor

Programming concepts explained through real-world architecture

Steps uses building architecture as a teaching tool. Students understand that buildings have floors, floors have rooms, and everything has a purpose and place. This same logic applies to programs in Steps.

  • Buildings - Complete programs with a clear entry point
  • Floors - Logical groupings of related functionality
  • Steps - Individual tasks that do one thing well
  • Risers - Hidden helpers that support steps internally

This metaphor isn't just cute - it fundamentally changes how students think about program organization.

üìñ English-Readable Syntax

Code that reads like instructions, not cryptic symbols

Students can read Steps code out loud and understand what it does. No need to memorize obscure symbols or wonder what == versus === means.

  • set age to 25 instead of age = 25
  • if score is greater than 90 instead of if (score > 90)
  • repeat 5 times instead of for(int i=0; i<5; i++)
  • for each item in list instead of obscure iterator syntax

This readability reduces cognitive load, letting students focus on algorithmic thinking instead of syntax memorization.

üß± Enforced Decomposition

One step per file prevents monolithic code

The biggest challenge in teaching programming is getting students to break problems into smaller pieces. Steps makes this mandatory, not optional.

  • One step = One file - Physical constraint forces modular thinking
  • Named steps - Every piece of functionality has a clear name
  • Visible organization - File system shows program structure
  • Reusable components - Steps can be called from anywhere

Students can't write a 500-line monolith. They must decompose. This builds the single most important programming skill.

calculate_tax.step 
step: calculate_tax
    belongs to: math
    expects: amount, rate
    returns: tax

    do:
        set tax to amount * (rate / 100)
        return tax

Each step is small, focused, and testable

üéì Educational Error Messages

Errors that teach, not frustrate

Most languages give cryptic error messages that leave students confused. Steps provides clear, educational feedback that helps students learn from their mistakes.

  • Plain English - No jargon or technical gibberish
  • Exact location - Shows exactly where the problem is
  • Explanation - Explains what went wrong and why
  • Solutions - Suggests how to fix the problem
  • Examples - Shows correct syntax when helpful

Students become independent problem-solvers instead of constantly asking for help.

üéØ Design Philosophy

Every feature serves the mission of teaching programming

Visible Structure

Program architecture is explicit in the file system, not hidden in code. Students can see organization at a glance.

Mandatory Decomposition

One step per file prevents monolithic code. Students must break problems down from day one.

Conscious Engagement

English-readable syntax requires understanding, not memorization. Students must think about what they're telling the computer.

Clear Data Flow

Explicit expects and returns declarations make data flow visible and intentional.

Educational Errors

Error messages explain what went wrong and how to fix it, turning mistakes into learning opportunities.

Progressive Disclosure

Simple programs are simple. Complexity is added gradually as students are ready for it.
